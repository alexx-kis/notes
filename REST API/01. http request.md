# HTTP запрос

## 1️⃣ Полная структура HTTP-запроса

```
POST /users HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVC...
Content-Type: application/json
Accept: application/json
Origin: http://localhost:3000

{
  "name": "Alex",
  "email": "alex@email.com"
}
```

🔹 1. Request Line

```bash
POST /users HTTP/1.1
```

Состоит из:

● Метод (POST)

● Путь (/users)

● Версия протокола (HTTP/1.1)

🔹 2. Headers

Самые важные:

`Content-Type`

Что ты отправляешь:

```bash
application/json
multipart/form-data
```

`Accept`

Что ты хочешь получить:

```bash
Accept: application/json
```

`Authorization`

```makefile
Authorization: Bearer <token>
```

`Origin`

Важно для CORS

🔹 3. Body

● Есть только у:

    ● POST

    ● PUT

    ● PATCH

● Обычно JSON

## 2️⃣ Полная структура HTTP-ответа

```pgsql
HTTP/1.1 201 Created
Content-Type: application/json
Cache-Control: no-cache

{
  "id": 42,
  "name": "Alex"
}
```

### 🔹 1. Status line

```
HTTP/1.1 201 Created
```

● 201 — создан ресурс

● 200 — просто успех

● 204 — успех без тела

### 🔹 2. Response Headers

    ● Content-Type

    ● Set-Cookie

    ● Cache-Control

    ● Access-Control-Allow-Origin

### 🔹 3. Body

● Может быть:

    ● JSON

    ● пустым (204)

    ● blob (файл)

## 3️⃣ Idempotency (очень важно)

Что это? Если повторить запрос несколько раз — результат одинаковый.

Метод Идемпотентен? GET ✅ PUT ✅ DELETE ✅ POST ❌ PATCH ❌

#### Пример:

```bash
PUT /users/1
```

● Если отправить 5 раз — пользователь всё равно один.

```bash
POST /users
```

● Если отправить 5 раз — создастся 5 пользователей.

## 4️⃣ CORS (глубокое понимание)

● Это механизм безопасности браузера.

Представь:

    Frontend: `http://localhost:3000`

    Backend:  `http://localhost:5000`

Это разные origin → включается CORS.

🔹 Что такое origin?

Origin =

```
protocol + domain + port
```

● Если хоть что-то отличается — это другой origin.

### Preflight-запрос (OPTIONS)

● Когда ты отправляешь:

```pgsql
POST с JSON
Authorization header
```

● Браузер сначала отправляет:

```bash
OPTIONS /users
```

● И спрашивает сервер: Можно ли мне сделать POST?

● Сервер отвечает:

```pgsql
Access-Control-Allow-Origin: http://localhost:3000
Access-Control-Allow-Methods: GET, POST, PUT
Access-Control-Allow-Headers: Content-Type, Authorization
```

● Если разрешено — браузер отправит основной запрос

● Preflight отправляется если:

    ● метод не GET/POST/HEAD

    ● есть кастомные headers (например Authorization)

    ● Content-Type не из safe списка
    (application/json уже вызывает preflight)

● То есть почти все реальные API-запросы с токеном вызывают preflight.

### 🧩 Реальный пример из Next.js

```ts
const createUser = async () => {
  const response = await fetch('http://localhost:5000/users', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      Authorization: 'Bearer token_here',
    },
    body: JSON.stringify({
      name: 'Alex',
    }),
  });

  if (!response.ok) {
    throw new Error('Request failed');
  }

  return response.json();
};
```

## Важные статусы

● 201 — создан ресурс

● 200 — просто успех

● 204 — успех без тела

● 401 - не аутентифицирован

● 403 - аутентифицирован, но нет прав/доступа

#### 🔥 409 Conflict

    ● создаёшь пользователя с уже существующим email

    ● пытаешься обновить устаревшую версию данных

#### 🔥 422 Unprocessable Entity

    ● данные прошли синтаксическую проверку

    ● но не прошли бизнес-валидацию

запрос:

```json
{ "email": "not-an-email" }
```

ответ:

```json
422
{
  "errors": {
    "email": "Invalid email format"
  }
}
```

👉 Middle frontend должен:

● правильно распарсить ошибки

● показать их в форме

#### 🔥 429 Too Many Requests

● rate limiting

● защита от брутфорса

● API лимиты

#### 🔥 500 vs 502 vs 503

● 500 сервер сломался

● 502 upstream сервер не ответил

● 503 сервис временно недоступен

👉 Frontend должен:

● показывать fallback UI

● возможно retry
